How It Works

The starter uses Spring Boot's auto-configuration. Add the dependency, set a single environment variable, and it wires the correct DataSource automatically.

1 Add Maven Dependency

Add the starter to your service's pom.xml:

<dependency>
    <groupId>com.asc.aava</groupId>
    <artifactId>aava-datasource-spring-boot-starter</artifactId>
    <version>1.0.1</version>
</dependency>

Ensure the AAVA feed is configured in your pom.xml or settings.xml:

<repository>
    <id>AAVA</id>
    <url>https://pkgs.dev.azure.com/ascendionava/AAVA/_packaging/AAVA/maven/v1</url>
    <releases><enabled>true</enabled></releases>
    <snapshots><enabled>true</enabled></snapshots>
</repository>

2 Exclude Spring Boot's Default Auto-Configuration

In your main application class, exclude Spring Boot's default datasource beans:

import org.springframework.boot.autoconfigure.jdbc.DataSourceAutoConfiguration;
import org.springframework.boot.autoconfigure.orm.jpa.HibernateJpaAutoConfiguration;

@SpringBootApplication(exclude = {
    DataSourceAutoConfiguration.class,
    HibernateJpaAutoConfiguration.class
})
@EnableTransactionManagement
public class MyServiceApplication {
    public static void main(String[] args) {
        SpringApplication.run(MyServiceApplication.class, args);
    }
}
Why? The starter provides its own DataSource, EntityManagerFactory, and TransactionManager. Without this exclusion, Spring Boot creates a conflicting second set and the app fails to start.

3 Add Properties in application.properties

Add the following properties to your service's application.properties. Each value uses ${ENV_VAR:default} syntax — the library picks the value from the environment variable first, and falls back to the default if not set.

NEW — These 2 properties are introduced by the datasource starter

# Base package(s) containing your @Entity classes (comma-separated for multiple)
aava.datasource.base-package=com.asc.aava.guardrails

# Auth mode — driven by environment variable, defaults to password
aava.datasource.auth-mode=${DATASOURCE_AUTH_MODE:password}

These are the only new properties you need to add. Everything else below already exists in most services.

# ─── Driver & Connection ───────────────────────────────────────────────
driverClassName=${DB_DRIVER:org.postgresql.Driver}
datasource.url=${DB_URL:jdbc:postgresql://aava-dev-postgres-db.postgres.database.azure.com:5432/postgres}

# ─── Password Configuration ───────────────────────────────────────────
# Option 1: Fetch password from Secrets Manager (Azure Key Vault / AWS Secrets / GCP Secret Manager)

# Option 2: Pass password directly via environment variable (fallback if secret not found)
datasource.password=${DB_PASSWORD:}

Password Resolution in password Mode

When auth-mode=password, the library resolves the DB password in this order:

1 Secrets Manager — If datasource.password.keyname is set AND a SecretsManager bean is available, it fetches the password from the vault (Azure Key Vault / AWS Secrets Manager / GCP Secret Manager)
↓ if secret not found or SecretsManager not configured
2 Environment Variable — Falls back to datasource.password which reads from DB_PASSWORD env var
Want to skip secrets and just use an env var for password?
Simply leave datasource.password.keyname empty and set the DB_PASSWORD environment variable directly. The library will use it as the connection password. 
# In your environment / deployment manifest:
DB_PASSWORD=my-actual-password

# In application.properties — leave keyname empty:
datasource.password.keyname=
datasource.password=${DB_PASSWORD:}
Want to use secrets manager (recommended for production)?
Set the secret key name and the library will fetch the password from your cloud's vault automatically: 
# The library looks up "DB-PASSWORD" key in your configured secrets manager
datasource.password.keyname=${DB_PASSWORD_KEY_NAME:DB-PASSWORD}

4 Configure Auth Mode Per Environment

Switch between modes by setting the DATASOURCE_AUTH_MODE environment variable — no code changes needed.

Password Mode (Local Dev / Traditional DB)

In password mode, the library needs a database password. You have two options to provide it:

Option A: Secrets Manager (Recommended for Prod)

Set the secret key name — the library fetches password from your vault automatically

datasource.password.keyname=${DB_PASSWORD_KEY_NAME:DB-PASSWORD}

Works with: Azure Key Vault, AWS Secrets Manager, GCP Secret Manager

Option B: Environment Variable (Simple)

Pass password directly via DB_PASSWORD env var — no secrets manager needed

datasource.password.keyname=
datasource.password=${DB_PASSWORD:}

Set DB_PASSWORD=yourpassword in your environment

# Full password mode example
aava.datasource.auth-mode=password
datasource.url=${DB_URL:jdbc:postgresql://aava-dev-postgres-db.postgres.database.azure.com:5432/postgres}
datasource.username=${DB_USER:postgres}
datasource.password.keyname=${DB_PASSWORD_KEY_NAME:DB-PASSWORD}
datasource.password=${DB_PASSWORD:}
Resolution order: The library first tries to fetch the password using datasource.password.keyname from the Secrets Manager. If the secret is not found or SecretsManager is not configured, it falls back to datasource.password (the DB_PASSWORD env var).

AWS IAM Mode (RDS — No Stored Passwords)

# application.properties
aava.datasource.auth-mode=aws-iam
datasource.url=jdbc:postgresql://my-rds-host.us-east-1.rds.amazonaws.com:5432/mydb
datasource.username=myiamuser
aava.secrets.region=us-east-1
aava.secrets.assume-role-arn=arn:aws:iam::123456789012:role/my-rds-role
No password needed. The library generates short-lived IAM authentication tokens (15 min TTL) via STS role assumption. Connections auto-refresh every 14 minutes.

Azure WIF Mode (Azure DB for PostgreSQL)

# application.properties
aava.datasource.auth-mode=azure-wif
datasource.url=jdbc:postgresql://myserver.postgres.database.azure.com:5432/mydb
datasource.username=my-aad-principal-name
No password needed. Uses DefaultAzureCredential which auto-resolves from Kubernetes WIF → Managed Identity → Azure CLI. Tokens refresh every 50 minutes.

GCP WIF Mode (Cloud SQL)

# application.properties
aava.datasource.auth-mode=gcp-wif
datasource.url=jdbc:postgresql://10.0.0.5:5432/mydb
datasource.username=my-sa@my-project.iam
Username format: Use the service account email without .gserviceaccount.com suffix.
Example: my-sa@my-project.iam for my-sa@my-project.iam.gserviceaccount.com
No password needed. Uses GoogleCredentials.getApplicationDefault(). Tokens refresh every 50 minutes.

5 Secrets Manager Configuration (For Password Mode)

If you use datasource.password.keyname to fetch the DB password from a vault, you need to configure the secrets manager connection in your consuming service's application.properties.

Required Dependencies (Only if using Secrets Manager)

Add these to your service's pom.xml — pick the adapter matching your cloud:

Required (Core)

<!-- AAVA Secrets Spring Boot Starter -->
<dependency>
    <groupId>com.asc.aava</groupId>
    <artifactId>aava-core-secrets-spring-boot-starter</artifactId>
    <version>1.0.5</version>
</dependency>

Pick Your Cloud Adapter

Add only the one matching your vault:

<!-- Azure Key Vault -->
<dependency>
    <groupId>com.asc.aava</groupId>
    <artifactId>azure-keyvault-adapter</artifactId>
    <version>1.0.5</version>
</dependency>

<!-- OR AWS Secrets Manager -->
<dependency>
    <groupId>com.asc.aava</groupId>
    <artifactId>aws-secrets-manager-adapter</artifactId>
    <version>1.0.5</version>
</dependency>

<!-- OR GCP Secret Manager -->
<dependency>
    <groupId>com.asc.aava</groupId>
    <artifactId>gcp-secret-manager-adapter</artifactId>
    <version>1.0.5</version>
</dependency>
Not using secrets manager? If you just pass the password via DB_PASSWORD env var directly, you don't need any of these dependencies. They are only required when you want the library to fetch the password from a vault.

Enable Secrets Manager

# Enable secrets integration
aava.secrets.enabled=${ENABLE_SECRETS:true}

# Enable managed identity (passwordless vault access)
aava.managed.identity=${ENABLE_MANAGED_IDENTITY:true}

Managed Identity = true (Passwordless — Recommended for Cloud)

When aava.managed.identity=true, the library connects to your secrets vault using managed identity / workload identity — no client secrets needed. You only need the vault endpoint:

Azure Key Vault

# Only vault URL needed
aava.secrets.vault-url=${AZURE_KEYVAULT_URL:}

Connects via Managed Identity / Workload Identity automatically

AWS Secrets Manager

# Only region needed
aava.secrets.region=${AWS_REGION:}

# Optional — only if cross-account
aava.secrets.assume-role-arn=${AAVA_ASSUME_ROLE_ARN:}

Connects via IAM role attached to pod/instance

GCP Secret Manager

# Only project ID needed
aava.secrets.project-id=${GCP_PROJECT_ID:}

Connects via Workload Identity / ADC automatically

Key point: With managed identity enabled, you do NOT need client-id, client-secret, access-key, etc. The cloud platform handles authentication automatically via the pod's identity.

Managed Identity = false (Service Principal / Static Keys — Local Dev)

When aava.managed.identity=false, the library authenticates to the vault using explicit credentials. Use this for local development or environments without managed identity.

aava.managed.identity=false
aava.secrets.enabled=true

# Azure Key Vault — full credentials
aava.secrets.vault-url=${AZURE_KEYVAULT_URL:}
aava.secrets.client-id=${AZURE_CLIENT_ID:}
aava.secrets.client-secret=${AZURE_CLIENT_SECRET:}
aava.secrets.tenant-id=${AZURE_TENANT_ID:}
aava.managed.identity=false
aava.secrets.enabled=true

# AWS Secrets Manager — full credentials
aava.secrets.region=${AWS_REGION:}
aava.secrets.access-key=${AWS_ACCESS_KEY_ID:}
aava.secrets.secret-key=${AWS_SECRET_ACCESS_KEY:}
aava.secrets.assume-role-arn=${AAVA_ASSUME_ROLE_ARN:}
aava.managed.identity=false
aava.secrets.enabled=true

# GCP Secret Manager — full credentials
aava.secrets.project-id=${GCP_PROJECT_ID:}
aava.secrets.credentials-path=${GCP_CREDENTIAL_PATH:}

7 Cloud Prerequisites

AWS — aws-iam Mode

  1. RDS instance must have IAM authentication enabled
  2. The IAM role must have rds-db:connect permission: 
    {
  "Effect": "Allow",
  "Action": "rds-db:connect",
  "Resource": "arn:aws:rds-db:<region>:<account>:dbuser/*/<username>"
}
  3. DB user must have rds_iam role: 
    CREATE USER myiamuser WITH LOGIN;
GRANT rds_iam TO myiamuser;

Azure — azure-wif Mode

  1. AAD admin must be configured on the PostgreSQL server
  2. DB username must match the AAD principal display name
  3. For Kubernetes: ensure AZURE_CLIENT_ID, AZURE_TENANT_ID, and AZURE_FEDERATED_TOKEN_FILE are injected via Workload Identity
  4. For local dev: run az login before starting the service

GCP — gcp-wif Mode

  1. Cloud SQL instance flag: cloudsql.iam_authentication = on
  2. Service account must have roles/cloudsql.instanceUser
  3. DB user created as IAM user: 
    CREATE USER "my-sa@my-project.iam" WITH LOGIN;
  4. For local dev: run gcloud auth application-default login

8 Remove Old Code (If Migrating)

If your service previously had inline datasource configuration, delete these files:

  • DatabaseConfig.java
  • DataSourceWithIAM.java
  • DataSourceWithAzureWIF.java
  • DataSourceWithGCPWIF.java

Also remove generic DB property-loading logic from your ApplicationPropertiesLoader — keep only service-specific secrets (Redis, API keys, etc.).

9 Verification Checklist

  • aava-datasource-spring-boot-starter:1.0.1 added to pom.xml
  • AAVA Maven repository configured in pom.xml or settings.xml
  • @SpringBootApplication excludes DataSourceAutoConfiguration + HibernateJpaAutoConfiguration
  • aava.datasource.base-package set to your service's entity package
  • aava.datasource.auth-mode set (or DATASOURCE_AUTH_MODE env var configured)
  • datasource.url and datasource.username configured
  • aava.datasource.schema set if not using default "core"
  • Old inline datasource config classes removed (if migrating)
  • Build passes: mvn clean package
  • Service starts and connects to DB successfully

10 Troubleshooting

No qualifying bean of type 'javax.sql.DataSource'
Cause: Missing exclusion on the main application class.
Fix: Add exclude = {DataSourceAutoConfiguration.class, HibernateJpaAutoConfiguration.class} to @SpringBootApplication.
Could not determine recommended JdbcType
Cause: Entity package not being scanned.
Fix: Set aava.datasource.base-package to the package containing your @Entity classes.
IAM token generation failed
Cause: Wrong AWS region or role ARN.
Fix: Verify aava.secrets.region and aava.secrets.assume-role-arn match your RDS instance.
Azure WIF token acquisition failed
Cause: Missing Azure identity environment variables.
Fix: Ensure Workload Identity env vars (AZURE_CLIENT_ID, AZURE_TENANT_ID, AZURE_FEDERATED_TOKEN_FILE) are injected, or run az login locally.
GCP WIF token refresh failed
Cause: No GCP credentials available.
Fix: Set GOOGLE_APPLICATION_CREDENTIALS env var or run gcloud auth application-default login.
Queries fail with schema error
Cause: Wrong schema configured.
Fix: Set aava.datasource.schema=your-schema-name in properties.
